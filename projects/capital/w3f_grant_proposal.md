## Project Details

### 1. 505 Systems — Democratic Power Contribution (DPC)

#### The Problem with Token-Weighted Governance

Token-weighted voting systems, including Polkadot's OpenGov, suffer from a structural flaw: power concentrates at the top regardless of contribution quality. A dormant whale who bought early but never shipped code, never participated in governance, and never contributed to ecosystem growth has significantly more influence than an active builder with modest holdings.

This creates three failure modes:
- **Participation theatre**: Builders participate knowing their votes barely register.
- **Plutocratic drift**: Governance outcomes reflect the preferences of early buyers, not active contributors.
- **Cartel formation**: Whales coordinate off-chain to capture governance before smaller participants realize the vote is already decided.

DPC does not replace token-weighted governance. It adds a second axis: contribution weight. The combination creates a system where both economic stake and demonstrated work earn influence — closer to how human organizations allocate decision-making power in practice.

#### DPC Mechanism

Governance weight `P` is derived from three verifiable dimensions:

```
P = f(Is, Ec, Rw)

Is = Structural Impact
    — verifiable contribution mass
    — on-chain actions: governance votes cast, code commits (verified via on-chain attestation),
      aid distribution events, validated community work
    — each action type has a defined base contribution score

Ec = Consistent Energy
    — temporal frequency dimension ("Proof of Grit")
    — recency-weighted: recent contributions matter more than historical contributions
    — decay function prevents historical whales from coasting indefinitely
    — active contributors maintain higher Ec than dormant ones

Rw = Weighted Resonance
    — peer review dimension
    — peer endorsements, weighted by the endorser's own DPC score
    — cartel detection: Truth-Anomaly Audit flags coordinated endorsement patterns
    — endorsements from high-DPC contributors carry more weight
```

**Key properties:**
- **Non-linear formula:** Cannot be gamed through a single dimension. A contributor who farms governance votes but produces no verifiable work will have high Is but low Rw. A well-endorsed contributor who never participates on-chain will have high Rw but low Ec. Maximum DPC requires consistent, multi-dimensional contribution.
- **Non-transferable:** DPC scores are account-bound and cannot be bought, sold, or delegated. Only earned.
- **Transparent and auditable:** All contribution data is on-chain. Any observer can verify why an account has a given DPC score.
- **Configurable hybrid:** Parachain teams can configure how DPC weight interacts with token weight — from pure DPC to pure token-weighted, with any blended ratio.

#### Technical Implementation — `pallet-dpc`

The Substrate pallet will expose the following storage items and extrinsics:

**Storage:**
```rust
// Contribution record per account
ContributionLedger: map AccountId => ContributionRecord {
    is_score: FixedU128,      // Structural Impact
    ec_score: FixedU128,      // Consistent Energy
    rw_score: FixedU128,      // Weighted Resonance
    dpc_score: FixedU128,     // Derived total
    last_updated: BlockNumber,
    contribution_count: u64,
}

// Endorsed relationships
EndorsementGraph: double_map (AccountId, AccountId) => EndorsementRecord {
    weight: FixedU128,
    block: BlockNumber,
    category: ContributionCategory,
}

// Configurable parameters (via governance)
DpcConfig: DpcParameters {
    is_weight: Perbill,   // Is component weight
    ec_weight: Perbill,   // Ec component weight
    rw_weight: Perbill,   // Rw component weight
    decay_rate: FixedU128, // Ec temporal decay rate
    cartel_threshold: Perbill, // Anomaly detection trigger
}
```

**Extrinsics:**
```rust
// Record an on-chain contribution
fn record_contribution(
    origin: OriginFor<T>,
    contribution_type: ContributionCategory,
    evidence: ContributionEvidence,  // IPFS hash of supporting data
) -> DispatchResult

// Endorse another account's contribution
fn endorse_contribution(
    origin: OriginFor<T>,
    target: T::AccountId,
    category: ContributionCategory,
    weight: Perbill,
) -> DispatchResult

// Oracle: attest external contributions (code commits, etc.)
fn attest_contribution(
    origin: OriginFor<T>,  // Oracle origin required
    target: T::AccountId,
    contribution: AttestationRecord,
) -> DispatchResult

// Query DPC score (read-only)
fn get_dpc_score(account: T::AccountId) -> FixedU128
```

**Runtime integration:**
`pallet-dpc` exposes a `DpcWeight` trait that `pallet-democracy`, `pallet-collective`, and custom governance pallets can implement to query contribution-weighted voting power as a complement to `pallet-balances`.

#### Integration with OpenGov

`pallet-dpc` is designed to integrate with Polkadot's existing `pallet-referenda` and `pallet-conviction-voting`. The hybrid voting weight formula:

```
final_weight = (token_balance * token_conviction * token_coefficient)
             + (dpc_score * dpc_conviction * dpc_coefficient)
```

Where `token_coefficient` and `dpc_coefficient` are governance-configurable parameters. A parachain wanting 50/50 hybrid governance sets both to 0.5. A parachain wanting pure DPC governance sets `token_coefficient` to 0.

### 2. Panik App — Decentralized Emergency Mesh

#### The Problem

Emergency response infrastructure is built on assumptions that fail the most vulnerable:
- Cell towers go down in disasters (when distress signals matter most)
- Centralized aid distribution enables corruption and diversion
- Government emergency systems are not available to people fleeing authoritarian regimes
- Activists, journalists, and refugees cannot use identity-linked emergency systems safely

**What fails:**
- SMS-based emergency systems: no cell coverage in disaster zones
- Centralized app-based systems: server goes down when infrastructure is damaged
- Identity-required systems: unsafe for people fleeing persecution

#### How Panik App Works

**One-button distress:** A single action triggers a distress signal — no account required, anonymous mode available.

**Mesh routing:** The signal propagates through a local mesh network:
1. **Bluetooth Low Energy (BLE):** Device-to-device propagation in dense areas
2. **Local WiFi Direct:** Higher bandwidth, longer range when available
3. **Satellite fallback:** Integration with low-earth-orbit satellite APIs when mesh coverage is insufficient
4. **Otto Device Mesh:** Purpose-built Otto hardware nodes deployed in high-risk areas (Wave II)

**Guardian Network:** Community members who opt in to receive distress alerts within a configurable geographic radius. Guardians earn contribution credit via 505 Systems DPC for verified response actions — creating an incentive layer for community safety participation without requiring financial compensation.

**ONEON Identity Integration:**
ONEON provides W3C DID-compatible identity for users who want to be known (registered guardians, aid organizations, verified responders) and privacy-preserving anonymous mode for users who cannot safely expose identity (activists, journalists, refugees in conflict zones).

**On-chain incident anchoring:**
Distress events, response actions, and resolution records are content-addressed (IPFS) and anchored to a Polkadot parachain. The on-chain record provides:
- Tamper-proof evidence for human rights documentation
- Auditable aid distribution records for NGO accountability
- Immutable timeline for post-incident investigation

#### Technical Stack

- **Mobile:** React Native (Android + iOS), targeting React Native 0.73+
- **Mesh protocol:** Custom BLE mesh layer built on top of React Native BLE PLX; WiFi Direct via native modules
- **Identity:** ONEON DID (did:oneon:) based on W3C DID Core 1.0 spec, Veramo SDK integration
- **Storage:** IPFS for incident records (content-addressed, pinned via Pinata/web3.storage)
- **On-chain anchoring:** Substrate-based parachain `pallet-incident-anchor` (lightweight, purpose-built for Panik incident records)
- **Guardian network:** Governed by `pallet-dpc` — guardian reputation is a DPC sub-score

#### Polkadot Parachain Module — `pallet-incident-anchor`

```rust
    timestamp: T::Moment,

    origin: OriginFor<T>,
    incident_id: T::Hash,

    responder_did: BoundedVec<u8, ConstU32<128>>,  // ONEON DID

// Resolve incident

    origin: OriginFor<T>,
    incident_id: T::Hash,

```

## Appendix A: DPC Mathematical Specification

### Contribution Scoring Function

Each contribution event `c_i` earns a base score `s(c_i)` based on contribution category:

| Category | Base Score | Attestation Required |
|---|---|---|
| Governance vote cast | 1.0 | On-chain (pallet-democracy) |
| Governance proposal submitted | 3.0 | On-chain |
| Code commit (verified) | 5.0 | Oracle attestation |
| Community work (validated) | 2.0 | Peer attestation (≥3 endorsers) |
| Aid distribution event | 4.0 | pallet-incident-anchor record |
| Guardian response (Panik) | 3.0 | pallet-incident-anchor record |

### Structural Impact Score (Is)

```
Is(account, t) = Σ [s(c_i) * recency_weight(t - t_i)]

recency_weight(Δt) = e^(-λ_Is * Δt)

λ_Is = 0.01 per epoch  // ~30 day half-life
```

### Consistent Energy Score (Ec)

```
Ec(account, t) = (active_epochs / total_epochs_since_first_contribution)
               * frequency_bonus(avg_contributions_per_epoch)

frequency_bonus(f) = min(1.0 + 0.1 * log(f + 1), 2.0)
```

Ec rewards sustained participation over time. A contributor active in 90 of 100 epochs scores higher than one who contributed heavily in 10 epochs.

### Weighted Resonance Score (Rw)

```
Rw(account) = Σ [endorsement_weight(endorser_i) * endorsement_strength_i]

endorsement_weight(endorser) = DPC(endorser) / max_DPC
                               * cartel_discount(endorser, account)

cartel_discount = 0.1 if cartel_detected(endorser_cluster, account)
                = 1.0 otherwise
```

Cartel detection: if ≥60% of endorsements come from accounts that mutually endorse each other at above-average rate, the cartel flag is set and endorsement weight discounted.

### Final DPC Score

```
DPC(account) = (w_Is * Is) + (w_Ec * Ec) + (w_Rw * Rw)

Default weights (governance-configurable):
w_Is = 0.5  // Structural impact primary
w_Ec = 0.3  // Consistency secondary
w_Rw = 0.2  // Peer recognition tertiary

DPC is bounded: [0, 100]
Normalization: DPC = 100 * raw_DPC / max_observed_DPC
```

## Appendix B: Panik App Offline Mesh Protocol

### Signal Propagation Algorithm

```
ONION-style relay with TTL limiting:

1. Originator generates DistressPacket:
   {
     incident_id: random 32-byte nonce,
     location_hash: HMAC-SHA256(lat_lng, secret_salt),  // privacy-preserving
     timestamp: unix_ms,
     anonymous: bool,
     did: optional ONEON DID,
     ttl: 5,  // max hops
     signature: optional (signed with ONEON DID key)
   }

2. Originator broadcasts via BLE advertisement (DistressPacket serialized as protobuf)

3. Relay node (guardian or bystander app) receives packet:
   - If incident_id seen before: DROP (deduplication)
   - If ttl == 0: DROP
   - Decrement ttl
   - Add relay signature (for response routing)
   - Re-broadcast on all available channels (BLE, WiFi Direct)
   - If guardian: ALERT user + store locally

4. Guardian response:
   - Guardian records response via ONEON DID
   - Response anchored to Polkadot parachain via pallet-incident-anchor
   - Response routed back to originator via relay chain (TTL-limited)
```

### Privacy Model

| Data | Stored on device | Stored on-chain | Visible to guardians |
|---|---|---|---|
| Distress signal | Yes (24h, then auto-deleted) | CID hash only | Yes (anonymized) |
| Location | Never stored in plaintext | Location hash (one-way) | No |
| Identity (non-anonymous) | Yes (ONEON DID) | DID only | Yes |
| Identity (anonymous mode) | No | No | No |
| Response record | Yes | Yes (CID) | Yes |
