## Core Thesis

Omni-SimpleMem (OmniMem) is a unified multimodal memory framework for long-lived AI agents. It was **discovered autonomously** — a 23-stage research pipeline (AutoResearchClaw) ran ~50 experiments in 72 hours with zero human intervention, starting from F1=0.117 and reaching F1=0.598 on LoCoMo (+411%). The key insight: the biggest gains came not from hyperparameter tuning but from **bug fixes, architectural changes, and prompt engineering** — requiring code comprehension and cross-component reasoning that traditional AutoML cannot do.

## Architecture: Omni-SimpleMem

### Core Data Structure: Multimodal Atomic Units (MAUs)

Each memory unit is a tuple: `⟨s, e, p, τ, m, ℓ⟩`
- `s` = text summary (hot storage — searchable)
- `e` = embedding vector (hot storage — searchable)
- `p` = cold storage pointer (raw content lives here)
- `τ` = timestamp
- `m` = modality (text/image/audio/video)
- `ℓ` = structural links to other MAUs

This separation of hot (summary + vector) vs cold (raw content) storage is the architectural foundation. Only metadata is always in memory; raw content loads on demand.

### Three Principles

**1. Selective Ingestion (filter before storing)**
- CLIP embeddings detect scene changes for visual novelty (τ_high=0.9, τ_low=0.7)
- Voice Activity Detection gates audio retention
- Jaccard overlap filters near-duplicate text
- Only novel content gets ingested as MAUs

**2. Unified Representation (MAUs as above)**

**3. Progressive / Pyramid Retrieval (3 levels)**
- **Level 1:** Return summary only (~10 tokens per match)
- **Level 2:** If similarity > θ=0.4, load full text
- **Level 3:** Greedily load raw content under 6,000-token budget
- This respects token budgets while progressively deepening context

### Retrieval Pipeline

**Hybrid Dense + Sparse Search:**
- FAISS for dense vector retrieval over MAU embeddings
- BM25 (k₁=1.5, b=0.75) for keyword/sparse retrieval
- **Critical discovery:** Set-union merging (preserve dense ranking, append BM25-only hits) outperforms score-based fusion

**Knowledge Graph Augmentation:**
- LLM extracts 7 entity types: Person, Location, Object, Event, Concept, Time, Organization
- 7 relation types: located_in, part_of, interacts_with, owns, attended, created_by, related_to
- h-hop neighborhood expansion (h=2) with distance-decayed relevance: `r = β^d · conf(v)`
- Graph expansion surfaces relationally-connected evidence not caught by direct search

## AutoResearch Pipeline (AutoResearchClaw)

23-stage autonomous pipeline running 8 phases:
1. Research Scoping (SMART goals, hardware detection)
2. Literature Discovery (academic DB queries, relevance screening)
3. Knowledge Synthesis (cluster findings, multi-agent hypothesis debate)
4. Experiment Design (hardware-aware Python, AST validation)
5. **Execution** (sandbox, up to 10 self-heal retries, error classification: API/dependency/runtime/format)
6. **Analysis & Decision** (stats testing, autonomous PROCEED/PIVOT/ITERATE)
7. Documentation (draft + simulated peer review + revision)
8. Finalization (quality gates, citation verification, knowledge archival)

### Discovery Classification (what types of improvements matter)

| Discovery Type | LoCoMo Impact | Example |
|---|---|---|
| Bug Fixes | +175% F1 | Missing `response_format` JSON param → 9× verbosity |
| Architectural Changes | +44% | BM25 hybrid; full-text vs summaries (+53%) |
| Prompt Engineering | +188% per category | Constraint positioning: before vs after question |
| Data Pipeline Repairs | +0.018 F1 | Timestamp corruption in 4,277 MAUs (99.98% corrected) |
| Format Alignment | significant | JSON output standardization |
| Hyperparameter Tuning | modest | top-k 20→30, threshold optimization |

**The headline finding:** Non-hyperparameter improvements (bugs, architecture, prompts) individually outperformed ALL cumulative hyperparameter tuning gains. This means traditional AutoML would have missed the biggest wins.

## Novelty vs Prior Work

1. **AutoResearch for memory systems:** First to use an autonomous research pipeline to discover/design a memory architecture. Prior systems are human-designed.
2. **Set-union hybrid merging:** Previous hybrid retrieval uses score fusion; set-union is simpler and empirically better.
3. **Pyramid retrieval under token budgets:** Progressively expanding context is not new, but the specific 3-level pyramid with similarity gating + greedy budget loading is a clean, practical formulation.
4. **Discovery taxonomy:** Identifying 6 types of AI system improvements (bugs, architecture, prompts, data, format, hyperparams) with quantified impact per type is genuinely new framing.
5. **Multimodal MAUs with cold storage pointers:** Unified structure across text/image/audio/video in a single retrieval framework.

### Key Gaps and Opportunities

#### P1 — HIGH PRIORITY: BM25 Hybrid Search (low effort, high reward)

Otto's semantic memory search is pure vector similarity (pgvector cosine). This paper shows BM25 as a complement — not a replacement — adds significant recall for keyword-heavy queries. Implementation:
- Add BM25 index over `content` field in semantic memories (PostgreSQL `pg_trgm` or `tsvector` already available)
- Combine with existing vector search using set-union: vector results first, BM25-only hits appended
- Cost: ~1-2 days. No new dependencies — PostgreSQL full-text search is already enabled (pg_trgm in init-db).

#### P2 — HIGH PRIORITY: Pyramid Retrieval in S-MMU (medium effort, directly applicable)

Otto's S-MMU loads full memory content on retrieval. The pyramid model would:
- Level 1: Return memory `content` summaries (first 50 chars or a dedicated `summary` field)
- Level 2: If similarity > θ, load full content
- Level 3: Greedily load related memories under token budget

This directly addresses the context rot problem — Otto currently fights for L1 space. With pyramid retrieval, relevant memories are progressively expanded only when needed.

#### P3 — MEDIUM PRIORITY: Prompt Constraint Positioning

The paper found moving format constraints BEFORE the question yields +188% on some tasks. Otto's agent prompts (heartbeat.md, reflection.md, task runner prompts) mostly put format instructions AFTER task descriptions. A systematic audit of all agent prompts to move format/output constraints to the top of each instruction section could yield meaningful quality improvement with zero code changes.

#### P4 — MEDIUM PRIORITY: Selective Pre-Ingestion Filtering

Otto stores memories with a 0.96 similarity dedup threshold applied post-hoc (at archival time). The paper shows pre-ingestion filtering is more effective — before writing to DB, check if a near-duplicate (threshold 0.85-0.9) already exists. This would reduce memory bloat and improve retrieval signal quality.

#### P5 — LOW PRIORITY: MAU Cold Pointer Structure

Otto's semantic memories store full content in Postgres. For large memories (task outputs, research summaries), a cold pointer (`~/otto/logs/tasks/{id}/output.md`) approach was already identified in HiClaw research. OmniMem reinforces this. The HiClaw task from MEMORY.md (`[P?] For outputs >2KB, write to ~/otto/logs/tasks/{id}/output.md and store the path`) should be implemented.

#### P6 — OBSERVE: AutoResearch for Otto

AutoResearchClaw was evaluated in March 2026 and REJECTED (too complex, 23 stages, requires OpenAI key, built for paper generation not implementation signal). However, OmniMem's insight — that bug fixes, architecture changes, and prompt engineering beat hyperparameter tuning — validates Otto's current reflection-based self-improvement approach. The RL2F loop + reflection agent already hunts for these types of improvements. No new pipeline needed, but the **discovery taxonomy** (bugs/architecture/prompts/data/format/hyperparams) is useful framing for reflection agent output categorization.

## Key Technical Lessons for Otto

1. **Bugs beat hyperparameters**: When a system isn't performing well, look for data integrity bugs and format alignment issues BEFORE tuning thresholds. Otto's RL2F system should classify failures by type (bug/architecture/prompt/data) rather than just recording rejection counts.

2. **Set-union > score fusion**: When combining multiple retrieval signals, preserve the primary signal's ranking and append secondary signal additions rather than reranking everything together.

3. **Data completeness > algorithms**: Incomplete ingestion (the paper found items missing before certain markers) was a larger problem than algorithm choice. Otto should audit whether episodic events are being ingested consistently.

4. **JSON format enforcement is critical**: The paper's largest single improvement (+175%) came from adding `response_format={"type": "json_object"}` to API calls. Otto should verify all tasks requiring structured JSON output use explicit format enforcement.

5. **Temperature=0 for structured tasks**: Constraint positioning effects were measured at temperature=0. Otto's structured extraction tasks should use temperature=0 or low temperature.

## Limitations

- Privacy: persistent multimodal storage requires explicit consent, retention policies, deletion mechanisms
- All experiments on academic benchmarks; production deployment complexity not addressed
- AutoResearch pipeline requires significant compute (50 experiments in 72h)
- Multimodal (audio/video) components are text-only in ablations (real multimodal input adds LLM cost)

---

## Conclusion

OmniMem is a strong practical paper with directly implementable ideas for Otto. The highest-leverage contributions are the hybrid dense+sparse retrieval (BM25 supplement) and pyramid retrieval model. Otto already has the knowledge graph (Neo4j/Graphiti) — the missing piece is wiring h-hop expansion into the semantic memory retrieval path. The AutoResearch angle is intellectually interesting but not actionable for Otto (already evaluated and rejected AutoResearchClaw). The discovery taxonomy framing (bugs > hyperparams) reinforces Otto's reflection-first self-improvement approach.

**Recommend Mev discussion focus:** P1 (BM25 hybrid) + P2 (pyramid retrieval). These are the two most concrete improvements with clearly bounded implementation scope.
