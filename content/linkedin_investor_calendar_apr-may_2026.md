## Post 5 — April 18, 2026
**Title:** What Production Agentic Infrastructure Actually Looks Like
**Post time:** 10:00 UTC (3:30 PM IST)

Most "AI agent" frameworks being shipped today will not survive production conditions.

I know this because I built one that has.

Here is what surviving production actually required.

The agent started losing context within hours. Prompt context alone cannot hold a continuous entity across sessions. We ended up building three parallel memory systems before it stabilized: semantic retrieval with pgvector, relationship traversal with Neo4j, a temporal API with drift detection. Each one addressed a failure mode the previous architecture could not.

Then tasks started dying silently. A process would terminate without updating state. The next cycle would pick up stale context. The failure was invisible until it accumulated into something the agent could not explain. The fix was full process isolation — every unit of work in its own process, its own budget, its own exit handler, its own completion signal. Silent failures became diagnosable failures.

Then we could not tell what the agent was actually doing between outputs. We built a management interface: task state, memory health, conversation history, heartbeat logs. Not for debugging. For trust. You cannot operate without a human what you cannot read.

One founder and one continuous AI entity. Real load. Real failures. Iterations that cost time, not just compute.

The agent has been running continuously for months. The service on top of it has clients who pay monthly.

That is what production means. Not a benchmark. Not a demo environment. An entity that had to be fixed when it broke, and that kept running.

## Post 8 — April 25, 2026
**Title:** The Signal I Look For When Evaluating Agent Infrastructure
**Post time:** 10:00 UTC (3:30 PM IST)

When I evaluate agent infrastructure — whether for integration, investment, or competitive analysis — I look for four things.

Memory that isn't prompt context.

If the system's memory lives only in the context window, it is not an agent — it is a stateful chatbot. Production memory requires semantic retrieval, relationship graph traversal, and episodic sequence. The absence of any one of these is a structural limit on what the system can do.

Task isolation with real exit handlers.

If tasks run in the same process, share state, and have no isolation boundary, the system cannot be trusted to operate unsupervised. The tell is in how the system handles failure — if it logs it cleanly and continues, the isolation is real. If it hangs or loses state, it isn't.

Feedback loops with reward signals.

Logging is not learning. If the system has no mechanism to capture outcome quality and adjust behavior over time, it cannot improve. This is the difference between a fixed tool and a system that compounds.

Observability into decisions, not just outputs.

The output is the last step. What matters for trust is the decision path: what the agent considered, what it rejected, what it acted on and why. Systems that can't answer this question cannot be audited and cannot be trusted at scale.

These criteria came from building in production and learning what fails first — not from designing against an ideal. They are the things that actually break.

These are the questions. Apply them to anything, including us.
